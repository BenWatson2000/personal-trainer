# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"Exervo" (formerly "My PT") — a vanilla-JS PWA personal trainer (12-week programme, meals, lifting log, progress tracking) hosted on GitHub Pages. **No frameworks, no build step, no bundler** — `index.html` + `app.js` + `styles.css` deploy exactly as committed. All user data lives on-device: localStorage (`pt_*` keys) is the source of truth, photos in IndexedDB (`pt-photos`). `sync.js` is an optional Supabase mirror that stays dormant until `SYNC_CONFIG` at its top is filled in.

## Commands

```bash
node --check app.js                      # syntax check (no linter configured)
python3 -m http.server 8000              # serve locally (any static server works)

# Full UI audit: 96 checks × 3 viewports (phone 390 / tablet 834 / desktop 1440)
npm i playwright-core                    # once; Chromium expected at /opt/pw-browsers/chromium-1194 (override: $CHROME)
node tests/gen-mock.cjs                  # regenerate tests/mock-data.json seed
node tests/ui-audit.cjs                  # runs everything, regenerates TEST.md

ONLY="E" node tests/ui-audit.cjs         # run a single group (prefix-matches the area name, e.g. A, B, … L1)

node tests/sync-flow.cjs                 # cloud auth+sync data-flow test (mock Supabase, no network)
```

`tests/sync-flow.cjs` runs the **real** sync.js/app.js against a mock supabase-js (route-swapped for `vendor/supabase-js.js`) with `requireAccount` active — asserting the sign-in gate, OTP code flow, upload payloads (per-key `user_state` rows with the authenticated `user_id`), pull-applies-remote, sign-out→gate, and account-switch wipe. It's fully offline/deterministic and runs in CI after the UI audit.

The audit is deterministic: it freezes the in-page clock to **Mon 2026-07-06 18:00** (Day 15, Wk 3) and seeds each session by importing `tests/mock-data.json` through the app's own Restore flow. **TEST.md is generated by the audit — never edit it by hand.** To cover a new feature, append an `["ID", "name", async (page, vp) => {…}]` item to the right group in `tests/ui-audit.cjs`.

## Shipping a change

1. Any change to `app.js` / `styles.css` / `sync.js` / `index.html` needs a **cache bump**: increment every `?v=N` in `index.html` and `sw.js`, plus the `CACHE = "pt-shred-vN"` name in `sw.js` (all must match, one `sed 's/v=60/v=61/g'` usually covers it). Without the bump, installed PWAs keep serving the old files.
2. Run the full audit; it must be 288/288 with an empty Findings section in TEST.md.
3. Opening a PR into `main` triggers the **UI audit** workflow (`.github/workflows/audit.yml`) — it installs Chromium, runs the full audit at `AUDIT_TIMEOUT=8000`, and fails the check on any finding (the audit exits non-zero; `resolveChrome()` finds the browser in CI). TEST.md is uploaded as the run artifact. Wait for that check to be green.
4. Squash-merge to `main` → verify the "pages build and deployment" workflow run is `success` (this is the deploy; there is no other pipeline).

## Architecture

**Rendering.** Each tab renders as one big template string into `#view` (`renderToday`, `renderShop`, `renderProgress`, `renderPhotosTab`, `renderSettings` in `app.js`, ~3000 lines, the whole app). A full `render()` kills the rest/interval/timelapse timers (`REST_INT`, `IV_INT`, `TL_INT`), so interactions inside the workout use **surgical DOM updates** instead: `applyMode()` (Gym↔Home swap via the shared `workoutRowsHtml()` builder), `updateWorkoutBadge/updateFuelBadge/updateDailyBadge` (fold summary counters), `updateRemaining()` (live kcal budget), `updateWater()`. `repaintKeepScroll()` is the safe default for everything else — it re-renders but preserves scroll position *and* `<details>` open state (`foldId()` keys folds by `data-panel` or digit-stripped summary text).

**Events.** One delegated click handler (`onViewClick`) dispatches on `data-act` attributes; there are no per-element listeners inside `#view`. Transient UI state lives in module-level vars (`CURRENT_TAB`, `OPEN_PANELS`, `OPEN_LIFT`, `OPEN_FORM`, `SWAP_SLOT`, `VIEW_OFFSET`, `HM_SEL`, `VIEW_PHOTO`) and is reset on tab switch.

**Content is data.** The entire programme lives in `data/plan.json`: `phases[].schedule` (7 days × gym `items` + `homeItems`), `meals` (14-day rotation), `mealBank` (Recipe Library pool), and `trainingPlans.gain` / `trainingPlans.older` (alternative programmes). Plan edits need no JS changes.

**Profile & goal engine.** `getProfile()` drives everything: `ageBand()` gates safety behaviour (under-13 blocked at onboarding, 13–17 run `numbersFree()` — no calorie numbers anywhere, cut coerced to maintain; 60+ gets `trainingPlans.older` and a relaxed calorie floor). Goal (cut/maintain/gain) sets direction via `dailyAim(pos)`, which layers: phase target → weight-recalc (`adjustedAim`) → treat payback (`paybackForDay`, spreads `pt_cheat_*` over 3 days) → adaptive tweak (`pt_adaptkcal`) → diet-break weeks (`pt_dietbreaks` → maintenance). `activePhases()` picks the training plan; meal portions scale to the aim with protein held (`pf`/`nf` factors).

**Day maths.** `position()` converts today's date + `pt_startDate` + `pt_shift` (push-back days) into `{dn, week, weekday, phase}`; per-day state keys are date-suffixed (`pt_checks_YYYY-MM-DD`, `pt_lift_…`, `pt_supp_…`, `pt_ready_…`, `pt_swap_…`, `pt_skip_…`, `pt_extra_…`, `pt_cheat_…`). `VIEW_OFFSET < 0` renders read-only recaps of past days.

**Sync (`sync.js`).** Wraps `localStorage.setItem/removeItem` to collect `pt_*` writes into a dirty queue, debounce-pushes to the Supabase `user_state` table (RLS: one row per user+key), and pulls anything newer on open/online via an `updated_at` cursor. **Require-account mode** (`SYNC_CONFIG.requireAccount`): `render()` shows `authGateHtml()` (a sign-in wall) whenever `ptSyncRequiresAuth()` is true — accounts enforced, no live session, and no `ptsync_account` flag. First use needs a connection; after signing in once the flag persists so the device works offline (session also persists). The gate is inert when sync is `unconfigured`, which is exactly the audit's state (`window.__PT_NO_CLOUD`) — so the offline audit still renders onboarding/app normally; if the gate ever leaks into the unconfigured path the whole suite fails at once. Magic-link email auth. "Clear all my data" calls `ptSyncOnReset()` so a device wipe is never mirrored to the cloud. Schema: `supabase/schema.sql`. Setup steps are in README. `SYNC_CONFIG` is **filled in and live** (project URL + `sb_publishable_…` key — the publishable key is safe to commit; never put the `sb_secret_…` key here). The supabase-js client is **vendored** at `vendor/supabase-js.js` (a self-contained UMD build exposing `window.supabase`, loaded by a plain script tag before app.js/sync.js and precached by the service worker) — **no runtime CDN dependency**, which is what makes sync reliable and offline-capable. `initSync()` uses `window.supabase.createClient`; the UI audit still sets `window.__PT_NO_CLOUD` so `initSync` bails before creating a client (no live network in the audit). Sign-in is **6-digit-plus code OTP** (`signInWithOtp`→`verifyOtp`, no magic-link redirect). Saving is automatic: `markDirty` debounces a `flush` ~700 ms after any `pt_*` write, and `online`/`visibilitychange` flush+pull on reconnect/refocus — there is no manual "sync now" button.

**Service worker (`sw.js`).** Network-first with cache fallback; the versioned cache name is what makes the bump ritual matter.

## Gotchas (each of these has already caused a real bug)

- The Up-Next hero pills carry `data-act="openpanel" data-panel="…"` — the **same `data-panel` attribute as the fold cards**, and they come earlier in the DOM. Any query for a panel card must use `details[data-panel="…"]`, never bare `[data-panel="…"]`.
- Toasts linger ~1.7 s. Tests must clear old toasts before triggering the one they want to read (`doToast()` helper in the audit).
- `#importFile` and `#photoUpload` are `display:none` inputs behind styled labels — Playwright must wait with `state: "attached"`, not the default visible.
- `#quickWeight` / `#logWeightBtn` ids appear in both the Today daily-log card and the Progress weight card. Only one exists per rendered tab, but don't add a third.
- `#syncEmail` / `#syncSend` (the magic-link sign-in) appear in both the onboarding sign-in card (`onboardSyncHtml()` in sync.js) and the Settings cloud-sync card — but onboarding and Settings are mutually exclusive screens (`needsOnboarding()` gates them), so only one instance is ever live. The shared `onViewClick`-independent document handler in sync.js drives both.
- Feedback rendered before a repaint dies with it — flash tips / open folds *after* `repaintKeepScroll()`, or rely on its fold-state preservation.
- Multi-column card layout (tablet ≥720px, `columns:2`) traps `position:fixed` — full-viewport overlays (photo lightbox) must be moved to the body-level `#overlay` div, which `render()` does automatically for `.lightbox`.

## Maintaining this file

Keep this document alive. When a session establishes a **durable rule** — a convention, a constraint, a gotcha discovered by hitting a bug — add it here in the same PR rather than leaving it as one-off chat guidance; that's the difference between advice that survives and advice that's lost when the session ends. Put new rules in the section they belong to (a shipping step under *Shipping a change*, a trap under *Gotchas*, etc.). Prune anything that's gone stale — a gotcha that no longer applies is worse than none, because it's trusted. This file is docs-only: editing it needs no cache bump.
